# 平台授权页 PRD

## 1. 需求判断摘要

- 场景：内容中心 / 平台授权页，用于管理可发布、可同步数据的第三方平台账号。
- 用户任务：运营或管理员需要添加平台账号、确认登录状态、查看账号归属信息，并在账号失效时处理。
- 用户决策链路：用户进入页面前知道需要发布内容或同步作品数据；进入页面后需要判断“哪些平台已授权、哪些账号可用、谁添加、是否需要重登或删除”。
- 当前阻塞：没有统一授权管理页时，发布任务无法判断目标账号是否可用，账号过期也缺少集中处理入口。
- 模块职责：建立平台账号可信状态，支撑内容发布和作品数据同步。
- 成功标准：用户可以在 1 个页面完成账号添加、筛选、测试登录状态和删除账号，并且每个操作都有明确反馈。
- 关键假设：第三方平台授权由后端统一接入，前端只承接平台选择、授权引导、状态展示和账号管理操作。

## 2. 需求结论

P0 版本必须完成平台账号管理闭环：

1. 支持添加平台账号。
2. 支持选择平台并获取登录码。
3. 支持扫码授权成功后的账号入表。
4. 支持平台、登录状态、用户名筛选。
5. 支持查看账号用户信息、添加人、添加时间、最近测试时间。
6. 支持测试登录状态。
7. 支持删除账号，且删除前二次确认。
8. 所有操作完成后必须给 Toast 提示。

P0 不保存真实密码，不在前端处理平台账号密钥；扫码登录、OAuth 回调、token 保存和状态校验均由后端负责。

## 3. 目标用户与核心场景

目标用户：

- 内容运营：负责多平台内容发布，需要知道哪些账号可用。
- 运营管理员：负责平台账号维护，需要添加、删除、测试账号。
- 研发 / 测试：验证后续平台授权接口和发布链路。

核心场景：

- 首次接入一个平台账号。
- 发布前检查账号状态。
- 平台账号登录失效后重新测试或删除。
- 多账号场景下快速筛选某个平台或某个用户名。

## 4. 模块定位与非目标

模块定位：

- 平台账号授权管理。
- 发布能力和数据同步能力的前置依赖。
- 第三方平台账号状态的可视化入口。

非目标：

- 不在本页创建内容发布任务。
- 不在本页展示作品数据。
- 不在 P0 实现账号权限分配到成员或角色。

## 5. 支持平台

P0 支持添加以下平台：

- 百家号
- 抖音
- 小红书
- 网易号
- 头条号
- 搜狐号

展示规则：

- 每个平台展示平台名称、简称标识和平台色。
- 同一平台允许添加多个账号。
- 平台列表由后端配置化返回，前端按返回的平台名称、平台编码、图标标识和平台色展示。

## 6. 用户流程

### 6.1 添加账号流程

1. 用户点击“添加平台账号”。
2. 系统打开添加弹窗。
3. 用户选择目标平台。
4. 用户点击“获取登录码”。
5. 系统展示该平台扫码登录码。
6. 用户扫码授权。
7. 平台授权成功后，后端完成账号绑定并返回授权结果。
8. 系统关闭弹窗，刷新账号列表，并将新账号按排序规则展示在列表中。
9. 系统 Toast 提示“平台账号添加成功”。

验收要点：

- 未选择平台时，“获取登录码”不可用。
- 获取登录码时展示 loading。
- 添加成功后不需要二次刷新列表。
- 新账号必须写入添加时间、添加人、最近测试时间。

### 6.2 查询筛选流程

用户可以通过以下条件查询账号：

- 用户名模糊搜索。
- 平台下拉筛选。
- 登录状态下拉筛选。

交互规则：

- 输入用户名后立即筛选。
- 选择平台或状态后立即筛选。
- 下拉筛选支持选项模糊搜索。
- 已选下拉条件支持点击叉号清除。
- 每次筛选条件变化后分页必须回到第一页。
- 查询无结果时展示空状态。

### 6.3 测试登录状态流程

1. 用户点击账号行内“测试登录状态”按钮。
2. 系统展示按钮 loading。
3. 后端校验平台授权状态并返回校验结果。
4. 最近测试时间更新为当前时间。
5. 系统按校验结果更新账号状态，并 Toast 提示结果。
6. 测试失败时应返回可读错误原因，例如 token 过期、平台接口异常、账号被封禁。

### 6.4 删除账号流程

1. 用户点击账号行内删除按钮。
2. 系统打开独立删除确认弹窗。
3. 用户点击取消，弹窗关闭，不改数据。
4. 用户点击确认删除，账号从列表移除。
5. 系统 Toast 提示“平台账号已删除”。

删除提示必须说明：

- 删除的平台。
- 删除的用户名。
- 删除后不能继续发布和同步数据。

## 7. 信息架构与展示规则

### 7.1 取数范围与统计口径

页面所有统计、筛选和列表数据均来自同一份账号集合。

生产环境取数：

- 数据源：平台账号管理服务返回的账号列表、统计数据和分页信息。
- 统计范围：当前企业下、当前用户有权限查看的全部平台账号。
- 筛选范围：平台、状态、用户名等筛选条件必须传给后端，由后端按统一口径返回结果。
- 删除后口径：账号被删除后，不从统计、筛选结果和分页总数中移除。停用账号可以正常统计
- 新增后口径：账号新增成功后，应刷新账号列表与统计数据；若当前筛选条件不匹配新增账号，则不展示在当前列表，但总体统计应更新。
- 时间口径：后端必须返回服务端时间或统一标准时间格式，前端不得使用浏览器客户端时间推断生产数据口径。
- 权限口径：后端必须按企业、角色、账号可见范围过滤数据，前端不得依赖隐藏字段实现权限隔离。
- 状态口径：账号状态由后端根据 token、平台返回码、授权有效期和内部规则计算，前端不得自行推断。

统计口径：

| 指标 | 计算规则 | 数据范围 | 排除规则 |
| --- | --- | --- | --- |
| 已添加账号 | `accounts.length` | 当前企业全部账号 | 已删除账号 |
| 登录状态正常 | `status === "正常"` 的账号数 | 当前企业全部账号 | 已删除账号 |
| 需要处理 | `status !== "正常"` 的账号数 | 当前企业全部账号 | 已删除账号 |
| 查询结果数量 | 命中当前筛选条件的账号数 | 当前企业全部账号 | 已删除账号 |

注意：

- 顶部统计卡默认不受查询区筛选条件影响，表达账号总体情况。
- 查询区右侧“共 N 个账号”受当前筛选条件影响，表达当前列表结果数量。
- 如果后续产品希望顶部统计也跟随筛选条件变化，必须在 PRD 中明确变更口径。

### 7.2 顶部统计卡

展示 3 个指标：

- 已添加账号：当前账号总数。
- 登录状态正常：状态为“正常”的账号数量。
- 需要处理：状态不是“正常”的账号数量。

### 7.3 查询区

查询区位于统计卡下方，列表上方。

字段：

- 用户名搜索框。
- 平台筛选。
- 状态筛选。
- 当前结果数量。

搜索框宽度应保持紧凑，不应撑满整行。

查询匹配规则：

- 用户名搜索：对 `username` 做模糊匹配。
- 平台筛选：精确匹配 `platform`。
- 状态筛选：精确匹配 `status`。
- 多个筛选条件之间使用 AND 关系。
- 筛选条件清空后，应立即按剩余条件重新计算列表。

### 7.4 列表字段

列表字段顺序：

1. 平台
2. 用户信息
3. 登录状态
4. 添加人
5. 添加时间
6. 最近测试时间
7. 操作

字段规则：

- 平台：展示平台图标标识和平台名称。
- 用户信息：展示头像和用户名。
- 登录状态：使用状态标签。
- 添加人：展示头像简写和姓名。
- 添加时间：格式为 `yyyy-MM-dd hh:mm:ss`。
- 最近测试时间：格式为 `yyyy-MM-dd hh:mm:ss`。
- 操作：仅展示图标按钮，包括测试登录状态、删除。

### 7.5 默认排序规则

默认排序：

1. 按 `addedAt` 倒序排序，新添加账号排在最前。
2. 如果 `addedAt` 相同，按 `lastTested` 倒序排序。
3. 如果时间仍相同，按 `id` 倒序作为稳定兜底。

筛选后的排序：

- 筛选只改变结果集合，不改变排序规则。
- 测试登录状态只更新 `lastTested`，不应改变默认排序，除非后续明确改为按最近测试时间排序。
- 新增账号插入后应出现在第一行，因为 `addedAt` 最新。

接口要求：

- 服务端分页时，排序必须由后端保证，前端不得在分页后再重新排序。
- API 需要明确返回排序字段和排序方向，例如 `sort=addedAt_desc`。

### 7.6 字段超长处理

超长展示规则：

| 字段 | 处理规则 | 悬浮/补充展示 |
| --- | --- | --- |
| 平台名称 | 正常展示；平台名称异常过长时单行省略 | title 展示完整平台名称 |
| 用户名 | 单行展示，超长省略 | title 展示完整用户名 |
| 添加人 | 姓名超长时单行省略 | title 展示完整姓名 |
| 添加时间 | 固定格式，不省略 | 无 |
| 最近测试时间 | 固定格式，不省略 | 无 |
| 删除确认文案中的账号名 | 超长时允许换行 | 保留完整文本 |

视觉要求：

- 表格行高不应因超长用户名无限增高。
- 账号名、平台名不得因为截断影响删除确认弹窗中的风险提示；弹窗应展示完整账号信息。

### 7.7 空数据、无结果与异常状态

空数据：

- 当账号总数为 0 时，列表区域展示“还没有添加平台账号”。
- 空状态应提供“添加平台账号”按钮。

筛选无结果：

- 当账号总数大于 0，但当前筛选无结果时，展示“没有找到符合条件的账号”。
- 文案建议提示用户清除筛选条件或修改用户名关键词。
- 无结果状态不展示添加按钮，避免误导用户认为没有任何账号。

加载状态：

- 获取登录码时，按钮展示 loading。
- 测试登录状态时，仅当前行按钮展示 loading，不阻塞整张表。
- 删除确认弹窗在提交删除后应展示删除中状态，并禁止重复提交。

异常状态：

- 获取登录码失败：停留在平台选择弹窗，Toast 提示失败原因。
- 测试登录状态失败：保持账号原状态，更新失败原因提示，不更新最近测试时间。
- 删除失败：账号保留在列表中，Toast 提示失败原因。
- 列表接口失败：展示错误态和重试入口。

低数据质量处理：

- 缺少 `addedAt` 时，添加时间展示 `-`，排序兜底到 `lastTested`。
- 缺少 `lastTested` 时，最近测试时间展示 `-`。
- 缺少 `addedBy` 时，添加人展示 `系统` 或 `-`，具体由后端口径决定。
- 缺少平台图标标识时，使用平台名称首字作为兜底。

### 7.8 分页

分页规则：

- 默认每页 10 条。
- 支持切换 20 条、50 条。
- 筛选条件变化时回到第一页。
- 表头吸顶，长列表滚动时保持列标题可见。
- 分页总数使用当前筛选结果数量，而不是账号总数。
- 删除当前页最后一条数据后，如果当前页无数据且前面仍有数据，应回退到上一页。
- 新增账号后，如果当前筛选条件匹配新增账号，列表应回到第一页并展示新账号；如果不匹配，则保留当前筛选结果并更新顶部统计。

## 8. 数据模型

```ts
interface PlatformAccount {
```

```ts
interface SupportedPlatform {
```

真实后端建议扩展字段：

- `platformCode`
- `accountId`

- `creatorId`
- `creatorName`
- `createdAt`

## 9. 状态规则

账号登录状态：

- 正常：账号可用于发布和同步。
- 即将过期：账号当前可用，但需要提醒重新授权。
- 登录失效：账号不可用于发布和同步，需要重新授权或删除。

状态变化：

- 后端必须根据 token、平台返回码和授权有效期计算状态。
- 前端不得自行推断真实授权状态。
- 新增账号后，状态以授权回调结果为准，通常为“正常”；若平台授权失败，则不得写入可用账号。
- 测试登录状态后，状态以校验接口返回为准。
- 删除账号后，账号从可用账号列表移除；历史发布任务是否保留账号快照按后端策略处理。

## 10. 系统职责边界

前端职责：

- 页面展示、筛选、分页、弹窗、Toast。
- 平台选择、授权引导和扫码登录结果展示。
- 展示账号状态、添加信息和测试时间。
- 展示空状态、loading 状态和删除确认。

后端职责：

- 返回平台账号列表。
- 返回支持平台列表。
- 生成平台登录码或 OAuth 授权地址。
- 接收授权回调并保存 token。
- 校验登录状态。
- 删除账号并处理发布任务引用关系。
- 返回可追溯错误码和错误原因。

规则引擎职责：

- 根据 token 过期时间、平台接口返回和内部阈值计算账号状态。
- 判断即将过期提醒窗口。
- 拦截登录失效账号参与发布任务。

AI / 模型职责：

- 本功能不需要 AI。
- 授权状态、时间、平台账号等关键事实必须由确定性接口和规则保证，不允许模型生成或推断。

## 11. 功能需求与验收标准

### FR-001 添加平台账号

需求：

- 用户必须可以点击“添加平台账号”打开弹窗。
- 用户必须可以选择支持平台。
- 用户必须可以获取登录码并进入扫码页面。
- 授权成功后账号必须加入列表。

验收：

- 未选择平台时不能获取登录码。
- 获取登录码期间显示 loading。
- 授权成功后列表首行出现新账号。
- 成功后 Toast 提示。

### FR-002 平台与状态筛选

需求：

- 用户必须可以按平台筛选。
- 用户必须可以按登录状态筛选。
- 下拉选项必须支持模糊搜索。
- 已选筛选项必须支持叉号清除。

验收：

- 选择筛选项后列表立即变化。
- 清除筛选项后列表立即恢复。
- 筛选后分页回到第一页。

### FR-003 用户名模糊搜索

需求：

- 用户必须可以输入关键词搜索用户名。
- 搜索框不应占满整行。

验收：

- 输入用户名片段即可匹配账号。
- 搜索无结果时展示空状态。

### FR-004 测试登录状态

需求：

- 每行必须提供测试登录状态按钮。
- 测试中必须有 loading 状态。
- 测试完成后必须更新最近测试时间。

验收：

- 点击后按钮进入 loading。
- 完成后账号状态为“正常”。
- 最近测试时间刷新。
- 出现 Toast 提示。

### FR-005 删除账号

需求：

- 每行必须提供删除按钮。
- 删除前必须二次确认。
- 删除后账号必须从列表移除。

验收：

- 取消删除不影响列表。
- 确认删除后列表数量减少。
- 删除后出现 Toast 提示。

### FR-006 添加人和添加时间

需求：

- 列表必须展示添加人。
- 列表必须展示添加时间。
- 新增账号时必须自动记录添加人和添加时间。

验收：

- 历史账号均有添加人和添加时间。
- 新增账号添加人为当前登录用户。
- 添加时间格式为 `yyyy-MM-dd hh:mm:ss`。

### FR-007 列表排序、边界与口径一致性

需求：

- 列表必须按 `addedAt` 倒序展示。
- 筛选结果必须保持默认排序。
- 顶部统计卡必须基于当前企业全部账号计算，不受查询区筛选影响。
- 查询结果数量必须基于当前筛选条件计算。
- 字段超长必须按规则省略或换行。
- 空数据、筛选无结果、接口失败必须展示不同状态。

验收：

- 新增账号后，如果命中当前筛选条件，应出现在列表第一行。
- 删除账号后，顶部统计、查询结果数量和分页总数同步减少。
- 筛选无结果时不展示添加按钮。
- 账号总数为 0 时展示添加入口。
- 超长用户名不会撑高表格行或挤压操作按钮。
- 缺少非关键字段时，页面按降级规则展示 `-` 或兜底标识。

## 12. 非功能需求

- 性能：筛选、分页、测试登录状态等操作应在接口正常响应后及时刷新；列表接口建议 P95 响应时间不超过 1.5 秒。
- 可用性：所有图标操作必须有 `title` 或 `aria-label`。
- 可读性：时间字段统一格式。
- 一致性：筛选组件、分页、Toast、表头吸顶必须复用全局组件和样式。
- 安全：不展示、不保存账号密码。

## 14. 质量门禁

上线前必须满足：

- 添加账号主链路可跑通。
- 删除弹窗必须独立展示，不复用添加弹窗状态。
- 所有列表操作完成后必须有 Toast。
- 筛选、分页、清除筛选必须联动正确。
- 登录失效账号不得被误判为可发布账号。
- 添加人、添加时间、最近测试时间不能为空。
- 删除账号前必须确认。
- 不允许保存或展示平台账号密码。
- 排序、统计、筛选、分页必须使用同一数据口径。
- 顶部统计和列表结果数量必须明确区分是否受筛选影响。
- 超长字段、空数据、筛选无结果、接口失败、操作失败必须有明确展示或提示。
- 服务端分页版本必须由后端保证排序，前端不得分页后重新排序。

## 15. 风险与取舍

风险：

- 真实平台授权流程差异较大，扫码登录、OAuth、Cookie 登录可能无法统一。
- 平台 token 过期规则不同，状态计算需要后端规则兜底。
- 删除账号可能影响历史发布任务和作品数据同步。
- 多账号同名时仅靠用户名不够，需要平台账号唯一 ID。

取舍：

- P0 不做账号权限粒度管理，后续再扩展账号可见范围和操作权限。
- P0 不展示授权能力字段，降低信息噪音。

## 16. 发布范围与降级方案

P0 发布范围：

- 支持平台账号添加、查询、测试状态和删除的生产链路。

降级方案：

- 平台登录码获取失败时，保留平台选择弹窗并提示失败。
- 登录状态测试失败时，保持原状态并展示失败原因。
- 账号列表为空时展示空状态和添加入口。
- 筛选无结果时展示无结果提示。

## 17. P1 / P2 规划

P1：

- 接入真实平台授权接口。
- 支持 token 过期提醒。
- 支持重新授权。
- 支持登录状态失败原因展示。
- 支持按添加人、添加时间筛选。

P2：

- 支持账号权限分配。
- 支持批量测试登录状态。
- 支持批量删除。
- 支持账号授权日志。
- 支持平台能力配置化。
