# GEO 双中心前端 Mock 方案

> 技术栈：Next.js + TypeScript + Tailwind CSS  
> 阶段目标：不接后端，先完成一套可操作、可演示、未来可平滑接入真实接口的前端产品原型。

## 一、需求判断摘要

### 1. 当前要验证什么

首版前端的价值不是把功能规划中的所有页面做齐，而是验证用户能否理解并完成这条核心链路：

> 从监测看板发现问题 → 查看原始回答证据 → 准备并发布内容 → 查看发布结果 → 创建复测 → 对比优化前后变化

### 2. 主要用户

- 品牌与市场负责人：关注品牌表现、竞品差距和优化结果。
- GEO / 内容运营人员：管理 Query、执行评测、发布内容和处理失败任务。

### 3. 首版成功标准

- 用户进入产品后，能快速理解“内容中心”和“监测中心”的职责。
- 用户能从看板指标下钻到具体 Query 和原始回答。
- 用户能完成一次模拟批量发布，并看到成功、失败和重试状态。
- 用户能从历史任务发起复测，并查看前后批次对比。
- Mock 数据和页面状态可替换为真实 API，不需要重写主要页面。

### 4. 关键取舍

- 保留规划中的 7 个功能入口，保证产品结构完整。
- 第一阶段只把一条核心闭环做深，其他页面先做到可浏览、可筛选和关键弹窗可操作。
- 不做登录、权限、多租户、真实文件上传、真实平台授权和真实模型调用。
- 不把缺失的平台指标展示为 `0`，统一展示“不支持”“未授权”或“暂未同步”。

## 二、产品信息架构

侧边栏只保留两个一级分组，不额外增加“评测结果”等重复入口。

```text
GEO
├── 监测中心
│   ├── 数据看板
│   ├── 任务管理
```

建议默认首页为 `/monitor/dashboard`。用户进入后先看到问题与机会，再进入内容发布，而不是先看到系统配置。

## 三、核心演示链路

前端第一阶段应优先保证以下演示路径完整，而不是平均建设所有页面。

### 链路 A：发现问题并查看证据

1. 用户进入数据看板。
2. 查看品牌提及率、推荐率、竞品答案份额和较上批次变化。
3. 在“待优化 Query”中选择一条品牌缺席或竞品领先的 Query。
4. 进入 Query 详情，查看不同 AI 模型的原始回答、品牌提及、推荐关系和引用来源。
5. 点击“为此 Query 准备发布内容”进入新建发布任务，自动带入关联 Query。

### 链路 B：模拟多平台发布

1. 用户填写标题、正文和封面占位图。
2. 选择发布账号，并预览不同平台版本。
3. 提交后进入发布任务详情。
4. 页面模拟任务从“等待中”变为“发布中”，最终产生成功和失败结果。
5. 用户查看失败原因并点击“重试”，状态更新为成功。

### 链路 C：复测与批次对比

1. 用户从任务详情或数据看板点击“创建复测”。
2. 新建评测任务自动继承 Query 集版本、模型和采样次数。
3. 模拟任务进度完成后进入批次对比。
4. 查看提及率、推荐率、竞品答案份额的变化，并下钻到发生变化的 Query。
5. 页面明确提示：变化表示相关性，不代表内容发布与指标变化之间存在确定因果。

## 四、页面方案

## 4.1 全局框架

### 页面结构

- 左侧：双中心导航，可折叠。
- 顶部：当前品牌切换、数据更新时间、通知入口、Mock 场景切换。
- 主内容区：页面标题、说明、主操作按钮、筛选条件和内容区。

### 全局 Mock 场景切换

顶部提供仅开发演示使用的“场景切换”：

- 正常数据
- 新用户空状态
- 部分任务失败
- 平台授权过期
- 数据暂未同步

这比为每种状态单独改代码更适合产品评审和演示。

## 4.2 数据看板

数据看板负责回答“现状是什么、问题在哪里、下一步做什么”。

### 信息优先级

1. **结果概览**：品牌提及率、推荐率、答案份额、引用来源数，展示较上批次变化。
2. **优化机会**：品牌缺席、竞品领先、错误描述、引用源机会。
3. **趋势与竞品**：批次趋势图、品牌与竞品答案份额对比。
4. **行动记录**：最近发布任务、最近评测任务和待处理失败。

### 关键交互

- 所有指标卡均可点击，打开对应 Query 明细列表。
- 图表点击具体数据点后，筛选对应批次和指标。
- 优化机会支持“查看回答”和“准备发布内容”。
- 支持选择两个批次进入对比页。

## 4.3 Query 管理与详情

### Query 列表

- 搜索、分类、标签、优先级、所属 Query 集筛选。
- 展示 Query、场景分类、优先级、最近提及结果、所属 Query 集。
- 支持新增、编辑、批量导入模拟、加入 Query 集。

### Query 详情

- 基本信息和所属 Query 集版本。
- 不同模型的结果矩阵：是否提及、是否推荐、提及竞品、状态。
- 原始回答抽屉或标签页。
- 引用来源列表，可标记来源类型。
- 历史批次变化。
- 主操作：“准备发布内容”“创建复测”。

## 4.4 品牌 / 产品管理

- 品牌卡片或表格：名称、官网、别名数量、产品数量、竞品数量、状态。
- 详情使用右侧抽屉，管理品牌介绍、别名、产品和竞品。
- 首版只在前端本地保存编辑结果。

## 4.5 评测任务管理与详情

### 任务列表

- 展示任务名称、Query 集版本、AI 模型、执行时间、进度和状态。
- 支持创建任务、查看详情、失败重试和创建复测。

### 任务详情

- 任务配置快照。
- 总数、成功、失败、剩余数量。
- 模拟执行进度。
- Query 执行明细和错误原因。
- 原始回答与结构化结果下钻。

## 4.6 发布任务与详情

### 新建发布任务

建议使用分步表单：

1. 填写内容：标题、正文、封面占位图、关联 Query。
2. 选择账号：选择多个平台和账号。
3. 平台预览：查看平台字段差异，允许手动调整。
4. 确认发布：汇总内容、目标账号和发布时间。

### 发布任务详情

- 总体进度与状态汇总。
- 各平台账号的发布状态。
- 平台返回链接、发布时间和失败原因。
- 单账号重试。
- 关联 Query 和“创建复测”入口。

## 4.7 平台内容

- 统一内容列表，按平台、账号、发布状态和时间筛选。
- 指标列展示浏览、评论、点赞和收藏。
- 缺失数据使用带原因的状态标签，不显示为零。
- 内容详情展示发布内容、平台链接、互动数据和最后同步时间。

## 4.8 平台授权

- 使用平台卡片展示支持的平台和能力范围。
- 展示正常、即将过期、已失效和异常状态。
- “授权”“重新授权”使用模拟弹窗和进度反馈。
- 明确标注支持发布和支持读取的指标。

## 五、视觉与交互方向

### 1. 整体风格

- 定位为专业的数据运营工作台，避免传统后台页面堆满表格。
- 浅色背景，白色内容卡片，使用单一品牌主色。
- 绿色表示改善或成功，红色表示失败或下降，橙色表示待处理；不能只靠颜色表达状态。
- 数据看板突出“问题与行动”，配置页面强调清晰和效率。

### 2. 通用组件

- `AppShell`：侧边栏、顶部栏和内容布局。
- `PageHeader`：页面标题、说明和主操作。
- `MetricCard`：指标、趋势、口径说明和下钻入口。
- `StatusBadge`：统一任务、授权、发布和同步状态。
- `DataTable`：筛选、排序、分页、空状态。
- `EvidenceDrawer`：查看原始 AI 回答和引用证据。
- `TaskProgress`：展示任务进度和失败数。
- `EmptyState`、`ErrorState`、`StaleDataNotice`：统一异常与降级表达。

### 3. 响应式范围

首版以桌面端为主：

- 主要适配宽度：`1280px` 及以上。
- `1024px` 可正常使用，侧边栏自动折叠。
- 手机端只保证基础浏览，不投入复杂表格和图表适配。

## 6.1 推荐依赖

| 能力 | 建议选择 | 用途 |
| --- | --- | --- |
| 框架 | Next.js App Router + TypeScript | 路由、布局和页面组织 |
| 样式 | Tailwind CSS | 视觉样式和响应式布局 |
| 基础组件 | shadcn/ui | 表单、弹窗、抽屉、表格基础样式 |
| 图标 | Lucide React | 导航和操作图标 |
| 图表 | Recharts | 趋势、份额和对比图 |
| 表单 | React Hook Form + Zod | 表单状态和校验 |
| 客户端状态 | Zustand | 模拟任务进度、筛选和跨页面状态 |
| 表格 | TanStack Table | 复杂筛选、排序和分页 |
| 日期 | date-fns | 时间格式化 |

### 为什么首版不直接使用 MSW

首版没有真实接口，也不需要验证网络协议。优先使用带类型的 Mock Service 层，开发更直接，也更容易维护交互状态。需要联调前，再在相同 Service 接口下切换到真实 `fetch`，或增加 MSW 做接口级模拟。

## 6.3 Mock 数据策略

### 数据原则

- Mock 数据围绕一个统一品牌案例构建，避免每个页面的数据互相对不上。
- 每条指标可以追溯到评测任务、Query 和原始回答。
- 发布任务必须关联 Query，复测任务必须关联历史评测任务。
- 平台指标需要包含“有数值”和“缺失原因”两种情况。

### 数据实体

```ts
type Brand = {};
type Product = {};
type Competitor = {};
type Query = {};
type QuerySet = {};
type EvaluationTask = {};
type EvaluationResult = {};
type PublishContent = {};
type PublishTask = {};
type PublishRecord = {};
type PlatformAccount = {};
type PlatformMetric = {};
```

正式实现时应为每个实体补充完整字段，避免使用页面专属的临时对象。

### 状态模拟

- Service 方法统一返回 `Promise`，默认延迟 `300-800ms`。
- 创建任务后由前端定时器模拟进度推进。
- 使用固定规则产生部分失败，便于演示重试。
- 编辑、新建和重试结果保存到 `localStorage`，刷新后仍保留。
- 提供“重置演示数据”操作，恢复初始场景。

### Service 接口示例

```ts
interface EvaluationTaskService {
  list(filters?: TaskFilters): Promise<EvaluationTask[]>;
  get(id: string): Promise<EvaluationTaskDetail>;
  create(input: CreateEvaluationTaskInput): Promise<EvaluationTask>;
  retry(taskId: string, queryIds: string[]): Promise<void>;
  createRetest(taskId: string): Promise<EvaluationTask>;
}
```

后续接后端时，仅替换 Service 实现，页面和组件继续使用同一接口。

## 七、开发阶段建议

### 阶段 1：产品骨架与核心数据

- 初始化 Next.js、TypeScript、Tailwind CSS 和基础组件。
- 完成全局布局、导航、页面标题和通用状态组件。
- 定义核心实体类型、统一 Mock 案例和 Service 接口。
- 完成数据看板静态布局。

**交付结果**：可以浏览完整导航，核心 Mock 数据关系成立。

### 阶段 2：监测中心闭环

- 完成数据看板交互和指标下钻。
- 完成 Query 列表、Query 详情和原始回答证据。
- 完成评测任务列表、任务详情、模拟进度和失败重试。
- 完成批次对比。

**交付结果**：可以演示“发现问题 → 查看证据 → 创建复测 → 对比结果”。

### 阶段 3：内容中心闭环

- 完成新建发布任务分步表单。
- 完成发布任务详情、模拟发布进度和失败重试。
- 完成平台内容列表和平台授权。

**交付结果**：可以演示完整 GEO 双中心闭环。

### 阶段 4：评审与打磨

- 补齐空状态、错误状态、过期状态和无权限占位。
- 增加 Mock 场景切换和演示数据重置。
- 检查页面跳转、数据一致性和可追溯关系。
- 优化桌面端布局、图表和表格体验。

## 八、首版验收重点

### 功能验收

- 7 个功能入口均可访问，不存在无响应的主要按钮。
- 核心演示链路可以从头到尾完成。
- 新建、编辑、重试和状态推进在刷新后可保留。
- 看板指标能够下钻到具体 Query 和原始回答。
- 发布任务与 Query、复测任务之间的关联清晰可见。

### 数据与内容验收

- 同一实体在不同页面中的名称、状态和指标一致。
- 看板汇总值与明细数据可以对应。
- 原始回答、引用来源和指标判断可以互相追溯。
- 缺失数据不展示为零。
- 批次对比不使用确定因果表达。

### 体验验收

- 用户进入产品后可以在 1 分钟内理解两个中心的职责。
- 页面主操作明确，每个页面最多突出一个主按钮。
- 所有异步操作都有加载、成功和失败反馈。
- 表格、图表、弹窗和抽屉视觉样式统一。

## 九、风险与待确认问题

### 主要风险

1. **页面范围过大**：若同时深做所有模块，容易变成大量低质量表格页面。应坚持先完成核心演示链路。
2. **Mock 数据割裂**：如果每页独立造数据，闭环无法成立。必须先定义统一实体和关联关系。
3. **只展示成功状态**：会导致产品无法验证失败重试、授权过期和数据缺失等关键体验。
4. **过早贴近后端设计**：当前阶段应稳定前端需要的 Service 契约，不必提前设计完整数据库和接口。

### 待确认问题

1. 首个统一演示案例使用哪个品牌、产品和竞品？
2. 首版计划模拟哪些 AI 平台和内容发布平台？
3. 前端更偏向用于内部产品评审，还是需要直接用于客户演示？
4. 是否需要同时提供深色模式、英文界面或移动端适配？

## 十、结论

这套前端可以直接使用 Next.js App Router、TypeScript 和 Tailwind CSS 开始开发。建议不要按菜单顺序逐页建设，而是先围绕统一 Mock 数据完成一条可操作的业务闭环，再补齐其他页面。

首个开发里程碑应是：

> 用户可以从数据看板发现一条品牌缺席的 Query，查看原始回答证据，创建并完成一次模拟发布任务，随后创建复测并看到批次对比结果。

达到这个里程碑后，前端已经能够验证双中心产品结构是否成立，也具备继续接入真实后端的基础。
